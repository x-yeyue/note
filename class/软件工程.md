# 第一章 软件工程概论

## 1.1 软件与软件危机

> **软件**是计算机程序、规程以及运行计算机系统可能需要的相关文档和数据。
	软件的真正含义不是一个形式的定义所能体现的。
> *从软件的内容*来看:
> 	软件更像是一种嵌入式的数字知识，其形式是一个通过 *交互对话* 和 *抽象理解* 而不断演化的过程。
> 
> **注:** 程序与软件不同，程序只是软件的组成部分。
> 	文档必不可少，只有程序不能称为软件。

一般可将软件分为 *系统软件*，*应用软件*，*可重用软件*。

### 1.1.1 软件的特点

**软件与硬件相比**的特点:
1. *软件是一种逻辑实体。*
	软件一般需要记录在专门的介质上，并且必须通过分析、测试和判断来了解它的功能、性能及其他特性。
2. *软件是人类的智慧结晶。*
3. *软件不会磨损，也不会被用坏。*
	软件不会像硬件那样磨损和老化，但会出现 *失效和退化* 的问题。如随着应用环境的变换，原来的软件不再适应新环境，因此需要进行维护。
4. *软件具有定制性。*
5. *软件成本相当昂贵。*
6. *软件维护工作复杂。*

### 1.1.2 软件的分类

按照 **功能** 的不同: 系统软件，支撑软件，应用软件。
	*系统软件* 
		计算机系统中最靠近硬件的，为其他程序提供低层的系统服务，它与具体的应用领域无关。
		例如: 编译程序、操作系统。
	*支撑软件*
		以系统软件为基础，以提高系统性能为主要目标，支撑应用软件的开发与运行。
		例如: 环境数据库、各种接口软件、工具组。
	*应用软件*
		提供特定服务的软件。
		例如: 字处理程序。
	注: 既有分工又有合作，不可分开的。

基于 **规模** 的不同: 微型、小型、中型、大型、超大型软件。
	*微型软件*
		例如: 某个学生为完成软件工程作业而写的一个程序。
	*小型软件*
		例如: 某个学生团队为完成软件工程的大作业(学期项目)而编制的程序。
	*中型软件*
		例如: 某个软件公司为某个客户开发的办公自动化系统(OA)。
	*大型软件*
		例如: 某个公司开发的某款多人在线网络游戏。
	*超大型软件*
		例如: 微软开发的 windows 10 操作系统。

根据 **软件服务对象** 的不同: 通用软件、定制软件。
	*通用软件*
		例如: 操作系统、文档处理系统、图片处理系统。
	*定制软件*
		例如: 企业定制的办公系统、交通管理系统、飞机导航系统。

按照 **工作方式** 的不同: 分时软件，实时软件，交互式软件，批处理软件

### 1.1.3 软件的发展过程

1. **程序设计阶段**(1946-1956 年)
	- 硬件: 电子管计算机 -> 电子管计算机。
	- 软件没有系统化的开发方法。
2. **程序系统阶段**(1956-1968 年)
	- 硬件: 晶体管计算机 -> 中小规模集成电路计算机。
	- 软件开发进入“作坊”式生产，出现了“软件车间”。
	- 出现大容量的存储器。
	- 出现高级程序设计语言。
	- 软件发展相对落后。
	- 导致了软件危机。
3. **软件工程阶段**(1968 年以后)
	- 硬件: 中小规模集成电路 -> 大或超大规模集成电路计算机。
	- 软件开发进入产业化生产，出现众多大型的“软件公司”。
	- 软件开发转化为工程方式。
	- 出现了“计算机辅助软件工程”。

### 1.1.4 软件危机

**软件危机** 是指在计算机软件的开发和维护过程中遇到的一系列严重问题。
	20 世纪 70 年代初爆发，至今软件危机仍未消除。

**软件危机的主要体现**：
- 如何开发软件，以满足对软件日益增长的需求；
- 如何维护数量不断膨胀的已有软件。

#### 软件危机的表现

1. 对软件开发成本和进度的估算往往不太准确。
2. 用户对“已完成的”软件系统的满意度较低。
3. 软件产品的质量较低，会经常出现问题。
   - 开发团队缺少完善的软件质量评审体系以及科学的软件测试规程，使得最终的软件铲平存在着诸多缺陷。
1. 软件尝尝是不可维护的。
2. 软件往往没有与配套相关的资料。
3. 软件成本在计算机系统总成本中占的比重逐年提高。
4. 软件开发生产率提高的速度，跟不上计算机应用普及深入的速度。

#### 软件危机产生的原因

1. 忽视软件开发前期的需求。
2. 软件不同于硬件，它是计算机系统中的逻辑部件而不是物理部件。
3. 软件不同于一般程序，它的一个显著特征是规模大，而且程序复杂性将随着程序规模的增加而呈指数上升。
4. 忽视于用户之间、开发组成员之间的交流。
5. 软件本身独有的特点确实给开发和维护带来一些客观困难。
6. 软件开发和维护有关的许多错误认识和做法的形成。
7. 没有完善的质量保证体系。

#### 解决软件危机的途径

学者从 *组织管理措施* 和 *技术措施*(方法和工具)两方面研究如何更好地开发设计和维护计算机软件，从而减低软件危机带来的影响。

1. 对计算机软件有一个正确的认识。
	- 彻底清除“软件就是程序”的错误观念。
2. 充分认识软件开发不是某个个体的神秘技巧，而是各类人员协作配合，共同完成的工程项目。
3. 推广和使用在时间中总结出来的成功软件开发技术和方法。
4. 开发和使用优质的软件开发工具。

## 1.2 软件工程概述

### 1.2.1 软件工程的三要素

> 软件工程以关注软件质量为目标，由 *过程*、*方法*、*工具* 三要素组成。

软件工程 **过程**: 
- 为了获得高质量地软件所需要完成的一系列任务的框架。
- 规定了 完成各项任务的工作步骤。
- 是*管理和控制产品质量的关键*。
- *定义了* 技术方法的采用、工程产品的产生、里程碑的建立、质量的保证、变更的管理。
软件工程 **方法**: 
- 完成软件开发的各项任务的技术方法。
- 为软件开发提供“如何做”的技术。
- *通常包括*: 某种语言或图形的模型表示方法、良好的设计实践、质量保证标准等。
- 使用*最广泛的两种方法*: 传统的软件开发方法、当下流行的面向对象方法。
软件工程 **工具**: 
- 为软件工程方法提供的 *自动* 或 *半自动* 的软件工程的支撑环境，辅助软件开发任务的完成。
- 现有的软甲工程*覆盖*了 *需求分析*、*系统建模*、*代码生成*、*程序调试*、*软件测试* 等方面，形成了 *集成化的软件工程开发环境* —— **计算机辅助软件工程**(Computer Aided Software Engineering, $CASE$)、

### 1.2.2 软件质量要素

> 国际标准化协会颁布的 $ISO \; 9126(2001)$ 将质量要素确定为 *正确性*、*可靠性*、*有效性*、*可用性*、*可维护性*、*可移植性* 6 个。
> 2011年3月，ISO 对 $IOS \; 9126$ 进行了修订和扩充，发布了 $ISO/IEC \; 25010$ 新标准，增强了 *安全性* 和 *兼容性*。

**软件工程8要素**: 
1. **正确性**
	软件满足需求规约和完成用户目标的程度。
2. **可用性** 
	学习和使用软件的难易程度，包括 *操作软件*、*为软件准备输入数据*、*结识软件爱你输入结果* 等。
3. **可靠性**
	软件完成预期功能，成功运行的效率。
	- 反映了软件 *无故障* 工作的状态。
4. **有效性**
	软件系统利用计算机的 *时间资源* 和 *空间资源* 完成系统功能的能力。
5. **可维护性**
	软件交付后的维护。
6. **可移植性**
	将软件安装在不同计算机系统或环境的难易程度。
	- 在软件设计过程中通常采用 *通用的程序设计语言* 和 *运行支撑环境*。
7. **安全性**
	控制或保护程序和数据不受破坏的机制。
8. **可复用性**
	*软构件* 在多种场合应用的程度。
	在更广泛的意义上，还应包括 *应用项目的复用*、*规约的复用*、*设计的复用*、*概念和方法的复用* 等。

**注**: 
	**软构件**: 概念或功能相对独立的一个或一组相关模块。

### 1.2.3 软件工程的目标和原则

软件工程的**基本目标**: 
1. 达到要求的软件功能。
2. 取得较好的软件性能。
3. 开发出高质量的软件。
4. 付出较低的开发成本。
5. 需求较低的维护费用。
6. 能按时完成开发工作，即时交付使用。

软件工程的**基本原则**: 
1. 用分阶段的生命周期计划进行严格的管理
2. 坚持进行阶段评审
3. 实行严格的产品控制
4. 采用现代程序设计技术
5. 软件工程结果应能被清晰地审查
6. 开发小组的人员应少而精
7. 承认不断改进软件工程实践的必要性
